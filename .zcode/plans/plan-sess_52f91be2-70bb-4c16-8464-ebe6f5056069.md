## Delete folder or file — in-memory, matching existing patterns

The client is fully in-memory (every mutation — `createCommit`, `moveNode`, `editFile` — is local React state, no backend). Delete matches that: a `deleteNode` state-update mirroring `moveNode`, triggered by a right-click context menu, guarded by a confirm dialog. No protocol/relay/Rust changes.

### 1. `deleteNode(path)` in `App()` — `apps/client/src/App.tsx`
Mirror `moveNode` (lines 1194-1247). For a file: drop that one key. For a folder: drop the folder, all descendant files (`path.startsWith(src + "/")`), and descendant empty-folders. Then clean up dependent state so nothing dangles:
- `setFiles` — remove deleted entries
- `setEmptyFolders` — remove the folder and descendants
- `setPanels` — any open panel whose path is the deleted file or under the deleted folder must be cleared (set to `""`, the existing empty-panel sentinel). **Critical:** `Panel` assumes `files[path]` exists (App.tsx:1317), so a dangling panel path would crash. `moveNode` rebases; delete must null out instead.
- `setCollapsed` — drop the folder and descendants from the collapsed set (harmless to leave, but tidy)

Use the existing `hasChild` helper (1249) to decide folder-vs-file scope, same as `moveNode` does.

### 2. Context menu — new component + state
- State in `Sidebar`: `ctxMenu: { x, y, path, type } | null` (type = "file" | "folder").
- `onContextMenu` handler on **both** the folder row div (App.tsx:230) and the file row div (App.tsx:294): `e.preventDefault()`, set the menu position to `e.clientX/clientY`, store the path + type.
- A positioned `<div className="ctx-menu">` rendered at the sidebar root, with one item: "Delete". Clicking it closes the menu and opens the confirm dialog for that path.
- Close on: click elsewhere, Escape, or scroll. (A single `useEffect` listening for `pointerdown` outside + Escape.)

Threading: add `onDeleteRequest(path)` to `TreeItem` props (lines 201-214), pass it from `Sidebar` (which owns the menu state). Mirrors how `onOpenFile`/`onToggleFolder` are already threaded.

### 3. Confirm dialog — new component + state
- State in `Sidebar` (or App): `confirmDelete: { path, type, count } | null`, where `count` = number of items affected (1 for a file, 1+N for a folder — computed by counting descendant files + empty folders, shown as "Delete X and N items?" when N>0).
- A small fixed-position overlay using existing CSS vars (`--surface-raised`, `--rule-strong`, `--accent`): message, "Delete" (accent/danger), "Cancel". No portal needed — a simple absolutely-positioned div with a backdrop.
- "Delete" → calls `onDelete(path)` (the prop from `App.deleteNode`) → clears confirm state.
- Folder case message: "Delete folder `<name>` and all its contents? This cannot be undone." File case: "Delete `<name>`? This cannot be undone."

### 4. Props threading
`App.deleteNode` → passed to `Sidebar` as `onDelete` → `Sidebar` wraps it to open-confirm-first, then passes `onDeleteRequest` to `TreeItem` (opens the context menu). Two layers:
- `TreeItem` calls `onDeleteRequest(path)` from the context menu → `Sidebar` opens confirm.
- Confirm's Delete button calls `onDelete(path)` → `App.deleteNode`.

### 5. CSS — `apps/client/src/App.css`
New classes, matching the existing mono/`--surface-raised`/`--accent` idiom:
- `.ctx-menu` — fixed/absolute, `--surface-raised` bg, `--rule-strong` border, mono font, z-index above tree rows, ~6px padding, each item a row with hover `--surface-sunken`.
- `.confirm-overlay` — fixed full-screen backdrop (semi-transparent `--ink`), centered.
- `.confirm-dialog` — `--surface-raised` card, padded, max-width ~320px, message + two buttons (Cancel = default, Delete = `--accent` or a new `--danger` if defined; otherwise `--accent`).

### Files touched
- `apps/client/src/App.tsx` — `deleteNode`, props threading, `onContextMenu` handlers, menu + confirm state/components.
- `apps/client/src/App.css` — `.ctx-menu`, `.confirm-overlay`, `.confirm-dialog`.

### Not doing
- No protocol/relay/Rust changes (in-memory only, matching `moveNode`/`createCommit`).
- No real filesystem deletion — consistent with the current app where nothing persists.
- No Rename in the menu (out of scope; the menu structure leaves room for it later).

### Verification
After implementing: right-click a file → menu appears → Delete → confirm → file gone from tree, no panel crash if it was open. Right-click a folder → confirm mentions contents → deleting removes folder + all children + closes any open panels inside. Escape/click-outside dismisses both menu and dialog.
