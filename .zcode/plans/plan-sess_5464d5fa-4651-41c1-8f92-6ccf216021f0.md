## What this adds

A **pin** = a piece of a file's content that is marked "settled." When you run `tracer run`, the LLM still produces a full-file rewrite (whole-document coherence preserved), but afterward the harness checks each pinned span against the model's output: if the model altered it, the canonical text is restored at its best-matched location; if it vanished entirely, that's surfaced as a conflict, not silently drifted. Over repeated rounds, more and more of the file becomes pinned, so the LLM is effectively only writing the unpinned remainder — sediment accrues until most of the file is settled.

The load-bearing decision (confirmed): **a pin is its canonical text, not its position.** Offsets are meaningless after any rewrite, so location is re-derived by matching text against current content each round. That is what lets pins survive arbitrary rewrites.

## Storage — local editor state, honors "no parallel content store"

A pin is per-file workflow state, in the same category as `registry.json` (attached folders) and `voices/`. It sits in `~/.tracer/pins.json`, managed by a new `PinRegistry` class mirroring `Registry`'s style (constructor reads file, methods persist on change). NOT in the relay, NOT signed — the "no parallel local store" prohibition (HANDOFF.md:25) was about edit *history*, and pins are not history.

```ts
// apps/harness/src/pins.ts
interface PinnedSpan { id: string; text: string; note?: string; pinnedAt: number }
interface FilePins { folderPath: string; relativePath: string; spans: PinnedSpan[] }
class PinRegistry {
  listPins(folderPath: string, relativePath: string): PinnedSpan[];
  addPin(folderPath: string, relativePath: string, text: string): PinnedSpan;
  removePin(folderPath: string, relativePath: string, pinId: string): void;
}
```

`text` is the canonical, immutable content of the pin. Identity is `(folderPath, relativePath, pinId)`. No offsets stored.

## Enforcement — post-write restore, inside `runLlmEdit`

Only the LLM rewrite path gets pin handling. `watch.ts` (external edits) and the test sites are untouched — pin enforcement is a property of *agent rewrites*, not of every snapshot, so it belongs in `agent.ts`, not pushed down into `recordSnapshot`. The clean seam.

New `apps/harness/src/pin-restore.ts` (pure functions, unit-testable without a relay):

```
restorePins(currentContent, modelOutput, pins):
  result = modelOutput
  conflicts = []
  for pin in pins:
    if modelOutput contains pin.text exactly:        # survived — leave it
      continue
    else:
      loc = bestMatch(pin.text, currentContent)      # locate pin in the PRE-rewrite file
      if loc.found:                                   # find nearest surviving anchor in modelOutput
        result = splice pin.text into result near loc.nearestAnchor
      else:
        conflicts.push(pin)                           # couldn't safely restore — surface, don't drift
  return { restored: result, conflicts }
```

Matching: exact substring first (`String.includes`). `bestMatch` uses `diff-match-patch`'s similarity (the dep already exists) to find the nearest anchor in the model output to the pin's original location, then splices the canonical text there. The pin's *canonical text* always wins over whatever the model produced for that span — that's the guarantee. If a pin can't be confidently relocated (heavy restructuring deleted both the span and its surroundings), it's a conflict, returned to the user, **not** silently dropped or silently drifted.

`runLlmEdit` in `agent.ts` changes to: reconstruct current content → fetch pins → ask provider for full rewrite → `restorePins(current, modelOutput, pins)` → write restored content to disk → `recordSnapshot(file, restored, 'llm', instruction)`. When `conflicts.length > 0`, still write the restored content (so progress isn't lost) but print the conflicts and set a non-zero / warning exit so the user notices. Record the prompt as before.

## CLI

- `tracer pin add <file> --text <span>` or `--lines <a-b>` — read file, extract span text, store as a pin. Pin is the *text*; `--lines` is just a convenience for selecting it.
- `tracer pin list <file>` — show pinned spans (id, first ~60 chars, note).
- `tracer pin remove <file> <pinId>` — unpin.
- `tracer run <instruction> -f <file>` — unchanged signature; pin enforcement now happens automatically when pins exist for the file. No new flag needed; a file with no pins behaves exactly as today.

## Protocol note (no wire change)

The enforcement outcome is *visible* to the user via CLI output, not encoded in the event. `recordSnapshot` already computes deltas against pre-rewrite content; restoring a pinned span before that call means the delta for a restored span is correctly a no-op (model changed it, we changed it back → net no change → no delta), which is exactly the desired provenance story: the pin's text is continuous across rounds. I will note this behavior in a short comment in `agent.ts` but will NOT add new protocol fields — `summary` exists in the spec for human-readable notes if we later want to record "N pins restored, M conflicts" on the event, but that's a follow-up, not this cut.

## What this deliberately does NOT do (deferred, per the user's "first cut")

- Pins are not signed/synced/composable (that's the deferred `sign` action, trace-provenance.md:181 — a real protocol change, out of scope here).
- No pin UI beyond the CLI (the Tauri/React client isn't built yet; when it is, pins surface naturally from `PinRegistry`).
- No automatic conflict-resolution UI beyond "print and warn" — restore-what-we-can, surface-the-rest.
